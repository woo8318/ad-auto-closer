# 전면광고 자동 닫기 앱 — 기획 정리

> 작성일: 2026-06-18
> 프로젝트 경로: `C:\Project\ad-auto-closer`

## 1. 목표 (무엇을 만들 것인가)

- **광고를 차단/스킵하는 게 아님.** 광고는 정상적으로 끝까지 본다.
- 전면광고(interstitial)가 **끝나서 닫기(X) 버튼이 나타나면**, 그 버튼을 **자동으로 눌러서 닫아주는** 안드로이드 앱.
- 즉, 동작은 "광고 닫기"가 아니라 **"닫기 버튼이 활성화될 때까지 기다렸다가 자동 탭"**.

### 왜 합리적인가
- 광고를 다 보고 나서 닫는 것이므로 광고 네트워크/개발자 수익에 영향이 거의 없다.
- 사용자는 매번 X를 손으로 누르는 번거로움만 제거한다.

## 2. 핵심 기술 원리

- 안드로이드의 **접근성 서비스(AccessibilityService)** 를 사용한다.
  - 원래 시각장애인 보조용 API. 화면의 UI 노드를 읽고(read) 대신 클릭(perform action)할 수 있다.
- 동작 흐름:
  1. 전면광고가 뜨면 처음엔 X가 없거나 비활성 (카운트다운 중)
  2. 광고가 끝나면 **X / "닫기" / "Skip" 버튼이 활성화**됨
  3. 접근성 서비스가 화면 노드를 스캔 → **클릭 가능한 닫기 버튼을 감지** → 자동 탭

## 3. 광고 생태계 현실 (설계 시 고려사항)

"전부 구글 광고(AdMob)"가 아니다. 여러 네트워크가 섞여서 나온다:

| 네트워크 | 비고 |
|----------|------|
| AdMob (Google) | 점유율 큼 |
| Unity Ads | 게임에서 특히 많음 |
| AppLovin / ironSource | 미디에이션에 자주 등장 |
| Meta Audience Network, Vungle 등 | 기타 |

- **미디에이션(mediation)**: 한 게임 안에서도 광고마다 다른 네트워크가 번갈아 뜬다.
- 따라서 **특정 네트워크 하나만 처리하면 안 되고, 범용 규칙**이 필요하다.

### 닫기 버튼 특성
- X 버튼 위치는 **거의 항상 화면 모서리(주로 우측 상단)** 에 고정 → 자동화에 유리
- X가 **글자가 아니라 아이콘(이미지)** 인 경우가 많음 → "Close 텍스트 찾기"만으로는 부족
- **contentDescription(접근성 라벨)** 이나 **위치 기반 탭**이 더 안정적

## 4. 감지 전략 (범용성 확보)

다음을 조합해서 닫기 버튼을 찾는다:

1. **텍스트 매칭**: "닫기", "Close", "Skip", "건너뛰기", "X", "✕" 등
2. **접근성 라벨(contentDescription)** 매칭: "close", "interstitial close button", "dismiss" 등
3. **위치 기반**: 화면 모서리(우측 상단 우선)의 클릭 가능한 작은 요소
4. **클릭 가능 여부(isClickable)** 와 **화면 표시 여부**를 함께 확인

## 5. 알려진 한계 / 변수

- **WebView/일반 UI로 그려지는 광고** (대부분의 전면광고): X가 노드로 인식되어 잘 동작함 ✅
- **게임 캔버스처럼 통째로 렌더링되는 광고**: X가 버튼/글자로 안 잡힘 ❌
  - 보완책: 정해진 화면 위치 탭 또는 **MediaProjection(화면 캡처) + 이미지 인식**
- 광고 제작자들이 의도적으로 만드는 방해 요소: 가짜 X, 지연 등장, 작은 버튼 → 100% 자동화는 어려움
- 오작동 방지: 게임 본화면의 X(예: 일반 닫기 버튼)를 광고로 오인해 누르지 않도록 **조건을 신중히** 설계해야 함

## 6. 구현 방향 (결정됨)

- **직접 앱 제작** 으로 진행 (MacroDroid 같은 기존 앱 사용 안 함)
- 언어/도구: **Kotlin + Android Studio**
- 핵심 컴포넌트: `AccessibilityService` 기반 백그라운드 서비스
