# 빌드 · 설치 · 테스트 워크플로우

> 로컬에 Android Studio가 없는 환경 기준.
> **GitHub Actions가 클라우드에서 APK를 빌드** → 갤럭시 S24에 설치해서 테스트한다.

## 전체 그림

```
[코드 작성/수정]
      │  git push
      ▼
[GitHub Actions]  ──► Gradle 빌드 ──► debug APK (artifact)
      │
      │  (APK 다운로드: S24 브라우저로 직접)
      ▼
[갤럭시 S24]  ──► APK 설치 ──► 접근성 서비스 활성화 ──► 테스트
```

핵심: **내 PC에는 아무 개발도구도 설치할 필요 없음.** 코드 편집 + git만 하면 빌드는 GitHub이 해준다.

---

## 1. 사전 준비 (1회)

### PC
- [ ] **Git** 설치 (코드 push용) — 이것만 있으면 됨
- [ ] GitHub 계정 + 새 리포지토리 생성 (이름 예: `ad-auto-closer`)
- (선택) GitHub Desktop 또는 VS Code의 Git 기능 사용 가능

### 갤럭시 S24
- [ ] **개발자 옵션** 켜기: 설정 → 휴대전화 정보 → 소프트웨어 정보 → "빌드번호" 7번 탭
- [ ] **출처를 알 수 없는 앱 설치 허용**: APK를 받을 앱(예: Chrome, 내 파일)에 권한 부여
  - (APK 설치 시 자동으로 권한 요청 화면이 뜨므로 그때 허용해도 됨)

> 갤럭시 S24는 Android 14(One UI 6.1). 접근성 서비스를 켜면 **"권한 제한됨/위험" 경고**가 나올 수 있는데,
> 사이드로드(스토어 외부 설치) 앱이라 그런 것. 직접 만든 앱이므로 허용하고 진행하면 됨.

## 2. GitHub Actions 빌드 파이프라인

리포에 `.github/workflows/build.yml` 을 두면, **push할 때마다 자동으로 APK가 빌드**된다.

빌드 결과물(APK)을 받는 방법 2가지:

### 방법 A — Actions Artifact (간단, 7~90일 보관)
1. GitHub 리포 → **Actions** 탭 → 가장 최근 빌드 실행 클릭
2. 하단 **Artifacts** 에서 `app-debug` 다운로드 (zip)
3. zip 안의 `app-debug.apk` 를 S24로 전송/설치
   - **팁**: S24의 Chrome으로 GitHub에 로그인 → Actions에서 바로 다운로드하면 PC 거칠 필요 없음

### 방법 B — Release 에 APK 첨부 (다운로드 링크 깔끔)
- 태그를 push하면 Release를 만들고 APK를 첨부하도록 워크플로우 구성
- S24에서 Release 페이지의 APK 링크만 누르면 바로 다운로드/설치

> 처음엔 **방법 A(Artifact)** 로 충분. 안정화되면 B로 전환.

### Debug APK 서명 관련
- `assembleDebug` 로 만든 **debug APK는 자동 생성되는 debug 키로 서명**되므로 별도 키스토어 설정 없이 바로 설치 가능.
- 나중에 정식 배포(Play 스토어 등)할 때만 release 서명용 키스토어가 필요.

---

## 3. S24에 설치

1. S24에서 APK 다운로드 (위 방법 A 또는 B)
2. 파일 탭 → APK 탭 → "설치"
3. "출처를 알 수 없는 앱" 경고가 뜨면 → 허용 → 설치 완료

---

## 4. 접근성 서비스 활성화 (필수)

앱을 설치만 하면 동작하지 않는다. **사용자가 직접 접근성 권한을 켜야** 한다.

1. 설정 → **접근성** → **설치된 앱**(또는 "다운로드된 앱")
2. `ad-auto-closer` 선택 → **사용 켜기**
3. "전체 제어 허용" 경고 → 확인

> 이 권한은 화면을 읽고 대신 클릭하기 위한 것. 앱의 핵심 동작에 반드시 필요.

---

## 5. 테스트 루프

1. 광고 나오는 게임/앱 실행
2. 전면광고가 떴다가 끝날 때 → **X 버튼이 자동으로 눌리는지** 확인
3. 로그 확인 (Android Studio 없이도):
   - 앱 내부에 **간단한 로그 화면**이나 토스트(Toast) 메시지를 넣어두면 "닫기 버튼 감지/클릭" 여부를 기기에서 바로 확인 가능 → 디버깅 편의상 초기 버전에 포함 권장
4. 안 닫히는 광고 발견 → 어떤 광고/네트워크인지 기록 → 감지 규칙 보완 → push → 재빌드 → 재설치

### 빠른 재설치 팁
- 같은 서명(debug)으로 빌드되므로 **기존 앱 위에 덮어쓰기 설치(업데이트)** 가능. 매번 삭제 안 해도 됨.
- 단, 접근성 권한은 업데이트 후에도 유지되지만, 가끔 꺼질 수 있으니 동작 안 하면 4번 재확인.

## 6. 주의 / 한계 (재확인)

- **게임 캔버스형 광고**: X가 노드로 안 잡혀 안 닫힐 수 있음 → 위치 기반 탭/이미지 인식으로 보완 (후순위)
- 오작동 방지: 게임 본화면의 닫기 버튼을 광고로 오인하지 않도록 감지 조건을 신중히 설계
